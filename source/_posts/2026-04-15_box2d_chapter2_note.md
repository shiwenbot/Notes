## 学习内容

### 1. WorldId 句柄设计

**核心概念**：
- WorldId 采用索引+版本号机制（Index1 + Revision）
- 避免"野指针"问题，失效的 ID 立刻可检测
- 支持多世界共存和确定性回放

**设计权衡**：
- 牺牲了 API 易用性（不能直接 `new World()`）
- 换来了数据导向的性能优势（连续数组、缓存友好）
- 多线程友好（静态方法更容易并行）
- 序列化简单（扁平数组 vs 对象引用图）

**学生理解**：
- 理解了"连续内存提高缓存命中率"的核心原理
- 理解了空间局部性原理（CPU 缓存行机制）

### 2. WorldDef 核心参数

| 参数 | 作用 | 调优场景 |
|------|------|---------|
| `Gravity` | 全局重力向量 | 横版游戏、太空游戏可调小或设为 0 |
| `ContactPushoutVelocity` | 事后修补穿模 | 穿模严重时提高，但太高会导致抖动 |
| `EnableContinuous` | 启用连续碰撞检测（CCD） | 预防高速物体穿模（隧道效应） |
| `EnableSleep` | 允许静止物体休眠 | 强烈建议开启，大幅提升性能 |
| `RestitutionThreshold` | 弹性生效的最低速度 | 堆叠物体多时降低可减少抖动 |
| `ContactHertz` | 接触约束刚度频率 | 数值越高接触越硬，但过高不稳定 |
| `MaximumLinearVelocity` | 刚体最大速度上限 | 防止隧道效应，默认 400m/s |

**学生理解**：
- 理解了穿模问题的两种解决思路：事后修补（ContactPushoutVelocity）vs 事前预防（EnableContinuous）
- 理解了"隧道效应"的概念
- 理解了参数调优的权衡关系（精度 vs 性能 vs 稳定性）

### 3. Step 机制（物理模拟的心跳）

**四阶段流程**：
1. **事件清空**：清空上一帧的事件数组
2. **BroadPhase**：用 BVH 树快速剪枝，排除不可能碰撞的物体对（O(n log n)）
3. **Collide**：精确检测 BroadPhase 筛选出的候选对（O(k)，k 是候选对数量）
4. **Solve**：速度积分 → 约束求解 → 位置积分

**时间步与子步**：
- `timeStep`：物理世界前进的总时间（建议固定，如 1/60f）
- `subStepCount`：把总时间切分成多少个子步（影响精度）
- 内部实际粒度：`h = timeStep / subStepCount`
- 黄金法则：**固定时间步长 + 适当子步数**

**学生理解**：
- 理解了 BroadPhase vs Collide 的区别（剪枝 vs 精确检测）
- 理解了为什么需要把碰撞检测分成两步（性能优化，从 O(n²) 降到 O(n log n + k)）
- 理解了 timeStep 和 subStepCount 的关系

### 4. Island 与图着色

**Island（岛屿）机制**：
- 把物理世界分成多个独立的"计算单元"
- 只计算活跃的 Island，静止的 Island 跳过
- 避免全局遍历，提升性能

**图着色（Graph Coloring）**：
- 把约束分组，同组约束可以并行求解
- 不同组依次求解，避免冲突（因为共享刚体）

**学生理解**：
- 知道了 Island 和图着色的存在
- 理解了"不需要手动管理 Island，引擎自动处理"
- 理解了这是引擎内部优化，用户层面不需要关心

### 5. 世界查询 API

**三个核心 API**：
- **`CastRay`**：射线检测，返回所有命中的物体
  - 回调返回 `-1`：忽略此结果，继续
  - 回调返回 `0`：立刻终止
  - 回调返回 `fraction`：截断射线，后续只搜更近的
  - 回调返回 `1`：不截断，继续
- **`CastRayClosest`**：快捷方法，直接返回最近的命中点
- **`OverlapAABB`**：AABB 重叠查询，快速筛选
- **`CastCircle`**：圆形投射，常用于技能判定

**QueryFilter 机制**：
- 通过位掩码过滤碰撞类别
- 过滤规则：`(shape.CategoryBits & query.MaskBits) != 0 && (shape.MaskBits & query.CategoryBits) != 0`

**学生理解**：
- 理解了 CastRay 回调返回值的机制
- 理解了如何用 `fraction` 实现"只找最近命中点"
- 理解了射击游戏子弹检测应该用 `CastRay` 并返回 `fraction`

### 6. 设计思考：ID + 静态方法 vs 传统 OOP

**Box2DSharp 的设计**：
- 不直接 `new World()`，而是 `World.CreateWorld(def)` 返回 `WorldId`
- 所有操作都是静态方法：`World.Step(worldId, ...)`

**优点**：
- ✅ 数据导向的性能（连续数组、缓存友好）
- ✅ 生命周期安全（ID 失效可检测）
- ✅ 多线程友好（静态方法更容易并行）
- ✅ 确定性模拟（序列化/回放简单）

**缺点**：
- ❌ API 不够直观（`World.Step(worldId, ...)` vs `world.Step()`）
- ❌ 不能直接在对象上"挂"自定义数据（需要维护 Dictionary）
- ❌ 调用更啰嗦

**学生理解**：
- 理解了"没办法携带很多对象封装好的变量"是不便之处
- 理解了这是"性能 vs 易用性"的权衡
- 理解了游戏引擎优先考虑"60 FPS 稳定性能"

## 教学反思

### 成功之处

1. **避免了"弱智问题"**：没有问"为什么必须是 World→Body→Shape 顺序"这种显而易见的问题
2. **深入设计权衡**：引导学生思考 WorldId 设计背后的性能考量（缓存、数据导向）
3. **联系实际场景**：穿模问题、射击游戏子弹检测、横版游戏重力设置
4. **苏格拉底式提问**：用问题引导推理，而不是直接给答案

### 需要改进的地方

1. **Island 和图着色**：学生表示"完全没接触过"，说明这个概念比较抽象
   - 改进：用游戏化的比喻讲解，而不是直接问技术细节
   - 结果：直接讲解后学生理解了"不需要手动管理"这个关键点

2. **CastRay 回调返回值**：学生一开始不清楚
   - 改进：用表格和具体例子说明不同返回值的行为
   - 结果：学生理解了如何用 `fraction` 实现"找最近命中点"

### 学生表现

- **理解能力强**：能快速抓住"连续内存提高缓存命中率"的核心原理
- **学习态度好**：主动指出之前的"弱智问题"问题，体现对教学质量的严格要求
- **实践经验丰富**：能联系 Unity 的 Rigidbody2D、Collider2D 等组件理解物理引擎
