## 核心概念速览

- **统一根类**：所有类型最终派生自 System.Object，保证统一头部格式
- **对象头部**：type object pointer（指向类型对象）+ sync block index（线程同步）
- **类型对象**：每种类型在堆上只有一份，包含静态字段和方法表
- **方法表**：扁平设计，子类复制父类所有虚方法槽位
- **虚方法调用**：运行时通过对象的 type object pointer 找到实际类型，再查方法表
- **is vs as**：is 检查两次类型，as 只检查一次更高效
- **Namespace**：编译器概念，CLR 不认识

## 恍然大悟时刻（重点！）

### 1. 为什么所有类型必须从 System.Object 派生？

**我的疑问**：C++ 不强制统一根类也能工作，为什么 .NET 必须这样？

**探索过程**：
- 三月七引导我思考 GC 如何识别堆上的对象
- 发现如果允许任意类定义，GC 无法确定"这块内存是不是对象"
- 统一根类保证了所有对象都有相同的头部格式（16字节：type object pointer + sync block index）

**最终解释**：
统一根类的核心价值不是提供 ToString/Equals 等方法，而是**保证运行时能统一管理所有对象**：
- GC 扫描时能识别对象边界
- 类型安全检查能追溯到根
- 反射能处理任何对象

**类比**：就像所有公民都有身份证（统一格式），政府才能建立统一的管理系统。

---

### 2. 多态的底层原理

**我的疑问**：虚方法调用时，CLR 怎么知道该调用哪个实现？

**探索过程**：
- 三月七用"查表"引导我思考
- 发现方法表存在类型对象里，不在实例对象里
- 关键问题：方法表是只存自己的方法，还是包括继承的方法？

**最终解释**：
- **方法表是扁平的**：子类方法表包含所有虚方法槽位（包括继承的）
- **type object pointer 在创建时确定**：`new Manager()` 时指向 Manager 类型对象，永不改变
- **虚方法调用步骤**：
  1. 通过引用找到堆上的对象
  2. 读对象头部的 type object pointer
  3. 找到类型对象的方法表
  4. 按方法槽位索引找到实现

**记忆点**：
- 非虚方法：编译时确定，直接调用
- 虚方法：运行时确定，"跟着引用走"

### 3. is + 显式转换 vs as

**我的疑问**：为什么要设计 as 操作符？

**探索过程**：
- 三月七让我比较两种方式
- 发现 is 检查一次类型，显式转换又检查一次
- as 合并成一次检查，失败返回 null

**最终解释**：
```csharp
// 方式1：检查两次（性能差）
if (o is Employee) {
    Employee e = (Employee)o;  // 又检查一次
}

// 方式2：检查一次（性能好）
Employee e = o as Employee;
if (e != null) { }  // 只检查 null，不检查类型
```

**记忆点**：as 转换失败不抛异常，返回 null。

---

### 4. System.Type 为什么要指向自己？

**我的疑问**：不懂，为什么不能指向 null 或另一个类型？

**探索过程**：
- 三月七反问：如果指向 X，那 X 的 type object pointer 指向谁？
- 发现会无限递归
- 如果指向 null，又破坏"万物皆有类型"的一致性

**最终解释**：
System.Type 是"类型的类型"，它描述自己。指向自己是唯一避免无限递归同时保持类型系统一致性的方法。

**类比**：字典里"定义"这个词的定义就是它本身。

**关键记忆点总结**：
1. 统一根类 → GC 和类型安全的基础
2. type object pointer → 对象的"身份证"，创建时确定永不改变
3. 方法表扁平化 → 空间换时间，虚调用只查一次表
4. as 优于 is+显式转换 → 少一次类型检查
5. System.Type 自引用 → 避免无限递归，保持类型系统一致性
