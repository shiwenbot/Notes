## 1. 动作切换的核心原则

### 1.1 角色同时只能处于一种动作

无论何时，角色同时只能处于一种动作。

两种触发方式：
- **玩家输入触发**：按Q → 放Q技能
- **环境交互触发**：攻击命中 → 受击者进入受击状态

**环境交互触发的优先级比玩家输入高**：确保控制技能能打断必杀技，维持游戏平衡性。

### 1.2 层级化管理动作

```
层级高 ── 特殊技能(Special Move) / 觉醒技能(OverDrive)
          ↑
层级中 ── 普通技能(Normal Move)
          ↑
层级低 ── 移动(Move)
```

**默认规则：低层级只能切换进高层级动作，同层级不能相互取消。**

特殊取消机制打破默认规则：
- **IASA帧取消**：收招动画部分可被任何动作取消
- **JC (Jump Cancel)**：技能被跳跃取消，跳跃优先级临时提高
- **加特林取消/挥空取消**等

### 1.3 取消与衔接的本质区别

| 类型 | 时机 | 说明 |
|------|------|------|
| **取消** | 动作执行**中** | 从某一帧启动取消窗口，立即跳转 |
| **衔接** | 动作执行**完毕后** | 下一帧立刻切换进下一个动作 |

取消的目的：**取消后摇，提高战斗节奏**。

## 2. 帧是动作游戏的逻辑单位

### 2.1 为什么用帧不用秒

**逻辑层认为帧间隔永远为16.666ms(1/60s)**，不管物理时间间隔如何变化。

- 用物理时间：30fps → 技能1秒=30帧，60fps → 技能1秒=60帧 → **不同设备逻辑不一致**
- 用逻辑帧：技能持续60帧，所有设备都是60帧 → **逻辑完全一致**

### 2.2 累加器机制

```csharp
while (self.Accumulator >= Dt)
{
    self.Accumulator -= Dt;
    self.Step();  // 执行一步逻辑帧
}
```

- **物理fps低（30fps）**：每帧33ms，够2帧逻辑 → **补帧**
- **物理fps高（144fps）**：每帧6.94ms，累积等待 → **跳帧**

## 3. 主时间轴 + 子时间轴

### 3.1 为什么需要多条时间轴

| 游戏机制 | 效果 |
|---------|------|
| **顿帧** | 攻击方和受击方动作暂停，特效/输入检测仍60fps |
| **子弹时间** | 怪物动作减缓，角色动作速度不变 |
| **时间静止** | 特定角色动作停止，其他逻辑继续 |

如果只有一条主时间轴，暂停=所有逻辑都暂停，**耦合严重**。

### 3.2 架构

```
游戏主时间轴（60fps，永不暂停）
├── SceneTimer（特效、输入检测...）
├── UnitTimer_A（角色A动作帧，可独立暂停/减速/加速）
└── UnitTimer_B（角色B动作帧，可独立暂停/减速/加速）
```

- **子时间轴由主时间轴驱动**：主时间轴每帧固定传入1/60s
- **Global.TimeScale**：影响所有逻辑（回放加速/减速）
- **Unit.TimeScale**：只影响单个Unit（顿帧、子弹时间）

## 4. 定时任务系统

> 动作帧中的"并行执行事件" = 定时任务：在时间轴某一帧注册，等待n帧后执行。

| 类型 | 说明 | 举例 |
|------|------|------|
| **OnceTimer** | 单次执行，执行后销毁 | 精神力炸弹延迟3帧引爆 |
| **RepeatedTimer** | 重复执行，自定义间隔 | 燃烧debuff每2帧触发 |
| **FrameTimer** | 每帧执行（RepeatedTimer特例） | 攻击判定持续期检测 |
| **OnceWaitTimer** | 等待（协程await） | await WaitFrameAsync(3) |

### OnceWaitTimer的本质

```csharp
async Task AnimCor(CancelToken token)
{
    UpdateAnim(1);
    await WaitFrameAsync(3, token);  // 启动OnceWaitTimer，3帧后唤醒协程
    if (token.IsCancel()) return;
    UpdateAnim(2);
}
```

### CancelToken的作用

动作切换时调用 `cancelToken.Cancel()` → 取消当前动作的所有等待任务。

**如果没有CancelToken**：技能被打断后，残留的定时任务仍然会执行 → bug！

## 5. 生命周期的执行顺序

```
每帧主循环：
├── 1. LogicUpdate（输入检测、角色逻辑）
├── 2. SceneTimer.Step()（主时间轴更新）
├── 3. UnitTimers.TimerUpdate()（子时间轴更新）
├── 4. PhysicUpdate
│   ├── PreStep（准备）
│   ├── Step（碰撞检测，缓存信息）
│   └── PostStep（处理碰撞回调）
└── 5. LateUpdate（相机、特效）
```

**LogicUpdate在PhysicUpdate之前**：即时反馈，当前帧更新判定框，当前帧就触发碰撞。

**PreStep → Step → PostStep**：Step()期间物理世界上锁，不能修改判定框。缓存碰撞信息到PostStep再处理。

## 6. 技能图 + 协程的组合方案

**技能图（graph/node）负责流程控制**：if判断、loop循环、分支

**协程负责具体动作执行**：await Frame_1、注册帧事件、更新判定框

**组合优势**：
- 技能图可视化编辑 → 策划也能配置
- 协程代码简洁 → 程序员易维护
- 各司其职，不冲突

## 面试要点

1. **为什么动作游戏用帧不用秒？** → 保证不同设备fps一致性
2. **累加器机制？** → 补帧/跳帧，保持逻辑帧间隔固定
3. **为什么需要主+子时间轴？** → 顿帧/子弹时间/时间静止，解耦
4. **取消vs衔接？** → 取消是执行中切换（取消后摇），衔接是执行完毕后切换
5. **CancelToken作用？** → 动作切换时取消所有等待任务，防止残留bug
6. **LogicUpdate为什么在PhysicUpdate前？** → 即时反馈，1帧延迟影响手感
