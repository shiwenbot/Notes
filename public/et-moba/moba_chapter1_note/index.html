核心概念速览#

  • CS代码共享：Unity客户端和.NET服务端共享同一份C#代码
  • 三层架构：Model（数据结构）、Hotfix（业务逻辑）、ModelView（表现层）
  • Link机制：服务端通过.csproj的<Compile Include>直接引用Unity工程中的代码文件
  • 条件编译：#if SERVER区分两端差异，客户端编译时完全不存在服务端专属代码
  • 客户端预测：双端共享战斗逻辑的核心原因——让客户端用相同逻辑做预测，提升操作手感

恍然大悟时刻#

1. 为什么要双端共享代码？#

我的疑问：服务端计算伤害不就行了吗？为什么客户端也要有相同的逻辑？

探索过程：

  • 三月七问：“你在玩LOL时，点击鼠标右键，角色是立即动还是等服务器回应？”
  • 我意识到：如果等服务器，网络延迟50ms，操作会卡顿
  • 三月七继续问：“放技能Q，伤害数字是立即显示还是等服务器计算完？”
  • 我明白了：客户端必须立即响应，但怎么保证和服务端一致？

最终解释： 这就是客户端预测！客户端用和服务端相同的代码预先计算：

  • 预测角色移动
  • 预测技能伤害
  • 预测Buff效果

等服务端结果回来后，如果一致就继续，不一致就回滚修正！

类比/记忆点：就像你在玩格斗游戏，本地立刻显示攻击动画，但实际伤害要等服务器确认。

2. Link共享机制是怎么工作的？#

我的疑问：服务端怎么"看"到Unity工程里的文件？会复制一份吗？

探索过程：

  • 我猜是用"include"方式引用
  • 三月七追问：..\..\Unity\Assets\Model\Core\**\*.cs是什么意思？
  • 我理解了：..\是相对路径，**是通配符

最终解释：

<Compile Include="..\..\Unity\Assets\Model\Core\**\*.cs">
  <Link>Core\%(RecursiveDir)%(FileName)%(Extension)</Link>
</Compile>
  • Include：源文件的实际位置（Unity工程）
  • Link：在Visual Studio里显示的虚拟路径（Core/）
  • 代码文件在磁盘上只有一份副本
  • 两个编译器各自编译，生成各自的DLL

类比/记忆点：就像Windows的快捷方式，文件只有一个，但可以从多个地方访问。

3. 三层架构的职责划分#

我的疑问：Hotfix是干啥的？Model就是纯数据，View就是纯UI表现？

探索过程：

  • 我理解了Model和ModelView，但不知道业务逻辑放哪里
  • 三月七举例：Buff系统
    • BuffData（数据）→ Model
    • Buff表现（图标、特效）→ ModelView
    • 但Buff的逻辑（每秒扣血、持续时间到了移除）放哪里？

最终解释：

层职责例子
Model数据结构、接口定义BuffData、NumericComponent
Hotfix业务逻辑实现Buff系统、技能系统、移动系统
ModelViewUI表现、渲染Buff图标、技能特效

共享方式：

  • Model层：完全共享（两端都需要数据结构）
  • Hotfix层：部分共享（战斗逻辑共享，UI逻辑客户端独有，数据库操作服务端独有）
  • ModelView层：客户端独有（服务端不需要渲染）

4. #if SERVER条件编译的作用#

我的疑问：条件编译是什么效果？编译进去但不执行？

探索过程：

  • 我以为：像if语句一样，运行时判断
  • 三月七纠正：这是编译时决定！
  • 三月七举例：Unity的#if UNITY_EDITOR，打包后这行代码的IL指令完全不存在

最终解释：

  • #if SERVER在编译阶段决定代码是否包含
  • 客户端编译时，#if SERVER包裹的代码完全不存在
  • 服务端编译时，#if SERVER包裹的代码才会被编译进去

三个使用场景：

  1. 服务端独有字段：帧快照字典（AttributeReusltFrameSnap）
  2. 服务端独有逻辑：Buff属性修饰器计算
  3. 客户端独有逻辑：渲染、表现代码（用#if !SERVER）

类比/记忆点：就像两家出版社从同一份原稿印书，A社印第1-10章，B社印第11-20章，各自印刷的内容完全不同。
