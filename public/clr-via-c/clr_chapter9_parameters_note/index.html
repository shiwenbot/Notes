CLR Chapter 9 - Parameters

日期: 2026-04-03 章节: CLR via C# Chapter 9 - Parameters 时长: 约50分钟（分两次课：9-1 + 9-2）

核心概念速览#

  • 默认参数版本陷阱: 编译器将默认值嵌入调用方 IL，只改定义不重编调用方会拿到旧值
  • 命名参数求值顺序: 语法可乱序，但编译器按物理位置从左到右求值
  • var 本质: 纯编译期类型推断语法糖，IL 中无 var；dynamic 是 Object + DynamicAttribute + 运行时绑定
  • ref/out 底层: IL 层几乎相同（仅 1 bit 差异），本质都是传地址
  • ref/out C# 语义: ref 要求调用方初始化；out 要求被调用方赋值，赋值前不可读取
  • ref 引用类型类型安全: 不允许 ref String 传 ref Object，防止方法体替换引用为不兼容类型
  • params 底层: ParamArrayAttribute，编译器在调用点自动构造数组
  • params 性能陷阱: 每次调用都有堆分配；高频场景用 1/2/3/4 参数显式重载优化（如 String.Concat）
  • 参数/返回设计原则: 参数用最弱类型（IEnumerable<T>），返回用最强类型（FileStream）

恍然大悟时刻（重点！）#

1. 默认参数的 “版本陷阱”#

我的疑问: 默认参数改了默认值，不重新编译调用方会怎样？

探索过程: 三月七拿 const 的跨程序集陷阱做类比。 varsity 都是"编译时 embedding"——const 是嵌入值，默认参数是嵌入默认值。

最终解释: 默认参数的默认值被编译器直接写进调用方 IL。如果只更新定义方法的 DLL，调用方 IL 里的旧值不会被更新。Richter 的解决方案：null sentinel + ?? 运算符，让运行时决定真正的默认值。

类比/记忆点: 和 const 是"双胞胎陷阱"，一个是嵌入常量值，一个是嵌入默认参数值。

2. 命名参数的求值顺序#

我的疑问: M(s: s_n++.ToString(), x: s_n++) 的输出是什么？

探索过程: 第一反应是按名字匹配顺序求值，答错了。后来意识到"物理位置左到右"才是关键。

最终解释: 命名参数只是调用语法层面的便利，编译器求值顺序雷打不动按代码的物理位置从左到右。有副作用表达式时这一点至关重要。

3. ref 引用类型的类型安全限制#

我的疑问: 为什么 ref String 不能传给 ref Object？

探索过程: 一开始以为是空引用问题，后来三月七追问"如果方法体把引用换成 Int32 会怎样"，才明白核心是"引用本身可被替换"。

最终解释: ref 允许方法修改引用本身的指向。如果允许 ref String → ref Object，方法体可以把 String 引用偷偷换成 Int32。调用方还以为是 String，访问时就变成了按错误偏移读取内存——不是 NullReferenceException，而是读到错误数据。这是 C# 编译器的有意保护。

解决方案: 泛型 Swap<T>(ref T a, ref T b) 同时保证类型安全和任意类型可用性。

4. params 的性能陷阱与框架优化#

我的疑问: params 方便是方便了，高频调用有什么问题？

探索过程: 三月七展示了 String.Concat 的重载列表——一大堆 1/2/3/4 参数的显式重载，最后才是 params 版本。

最终解释: params 本质上是编译器在调用点帮你 new 一个数组。每次调用都要堆分配、初始化元素、最终 GC。String.Concat 的 1~4 参数重载完全避免了数组分配，是高频场景下非常经典的优化模式。这个思路可以直接迁移到自己的 API 设计中。

5. 参数与返回类型的设计原则#

我的疑问: 参数和返回类型应该选强类型还是弱类型？

最终解释:

  • 参数用最弱类型：IEnumerable<T> 比 List<T> 灵活，能接受数组、List、HashSet、字符串等所有可遍历对象
  • 返回用最强类型：返回 FileStream 比 Stream 给调用方更多选择（需要 Stream 时自己向上转型即可）；返回弱类型会剥夺调用方访问子类特有方法的权利
  • 例外：当你想保留未来修改内部实现的灵活性时，返回可以用 IList<T> 这种"中等强度"的类型
